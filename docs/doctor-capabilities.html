CLI / DOCTOR

Doctor 是工作台体检单

starwork doctor 回答两个问题：这个 StarWork 工作台是不是完整；如果是旧目录，它有哪些可以整理成 StarWork 的线索。

健康检查检查工作台身份证、入口文件、场景能力、正式成果区和日常工作区。
历史信号识别旧目录线索，推断语言、用途和已有目录分工。
人类报告默认输出给普通用户看的体检结果和结构风险。
机器输出--json 输出目录清单和整理线索，给 skill、脚本和后续 repair 使用。

两种使用场景

同一个命令，面对新 StarWork 工作台和历史模板时，回答的问题不一样。

标准工作台

检查是否健康

当目录里有 .starwork/workspace.json，doctor 会把它当成标准 StarWork 工作台，检查工作台身份证、入口文件、场景能力、正式成果区和日常工作区。

  • 初始化后检查。
  • 安装 Pack 前后检查。
  • Agent 接手项目前先跑一次。
历史模板

暴露整理线索

当目录没有工作台身份证，但有旧模板的痕迹，doctor 会识别它是不是旧目录候选，并把目录结构和候选语义交给 starworkDoctor 判断。

  • 适合已经在使用旧模板的学员。
  • 适合用户不确定自己目录结构属于哪一类。
  • 适合后续设计 upgrade 或 repair 命令。

执行步骤

Doctor 的顺序很克制：先确认身份，再检查结构，最后只暴露事实和信号。

01定位目录

从 --target 或当前目录开始查找工作台根目录。

02识别状态

优先读取 .starwork/workspace.json；没有工作台身份证时进入旧目录诊断。

03检查结构

检查入口文件、关键目录、场景路径、初始文件和模板文件。

04输出分级

每个检查项给出 pass、info、warn 或 fail。

05暴露信号

标准工作台给健康结论；旧目录给目录清单、线索和推测。

检查范围

当前 doctor 以确定性结构检查为主，避免对用户内容做主观判断。

范围检查什么常见失败原因
工作台身份证工作台版本、类型、语言、场景能力列表、正式成果区、日常工作区。缺少 .starwork/workspace.json，或里面缺少关键字段。
入口和关键文件AGENTS.md、项目状态、当前工作、正式成果区、日常工作区。AI 入口规则被删，项目状态或当前工作文件缺失。
基础结构基础结构源目录是否存在，工作区类型和基础结构是否匹配，必要文件是否完整。工作台身份证里的类型写错，或误删必要文件。
场景能力场景能力是否适合当前工作台，相关路径、初始文件和模板是否落地。场景初始文件被删，或场景能力不支持当前工作台类型。
定制方案定制工作台的方案格式、定制目录、初始文件、注入规则是否存在。定制项目生成后，用户手动删除了定制目录或 AGENTS 规则块。
旧目录历史模板里的入口规则、系统目录、事项、参考资料、输出、身份、教训。目录可识别但缺少工作台身份证，需要先升级或补齐。

结果分级

分级决定用户看到的严重程度，也决定命令退出码。

级别含义是否影响退出码
pass检查通过。不影响。
info提供上下文信息，例如推断出的语言、工作区类型、参考资料目录。不影响。
warn有风险，但工作台仍可继续使用，例如历史模板缺少 AI 入口规则。默认不影响；--strict 下影响。
fail缺少关键结构或存在阻塞问题，例如没有工作台身份证、缺少正式成果区、场景初始文件丢失。影响，退出码为 1。

历史模板升级诊断

这是当前最重要的新能力：帮已有历史模板的用户找到温和迁移路径。

识别信号

  • AGENTS.md、CLAUDE.md、.cursorrules
  • _系统/、_system/
  • 事项/、matters/
  • 参考资料/、references/
  • 输出/、outputs/
  • identity/、lessons/

推断逻辑

  • 中文路径更多，推断为 zh；英文路径更多，推断为 en。
  • 存在 事项/ 或 matters/ 时，只记录为历史内容信号。
  • 非 Hub 候选统一推断为 project。
  • 不直接判断场景能力贴近度；场景能力只在后续执行层作为候选参数。
关键边界 Doctor 不会自动改历史模板。它负责把目录结构和候选信号暴露出来；starworkDoctor skill 负责结合上下文做判断，并在用户确认后生成升级方案。真正写入仍由 starwork upgrade --blueprint 执行。

Doctor 与 Skill 的分工

平滑升级需要理性判断，所以 CLI 和 AI Skill 必须分层。

Doctor

探测当前目录，输出工作台身份证、目录清单、线索、检查结果和旧目录候选信号；不输出下一步建议。

starworkDoctor

基于 doctor 的事实，判断当前目录分工是否清楚；用户要求升级时，继续确认目录语义并生成升级方案。

Upgrade

在用户确认后执行升级方案中的低风险写入；这里才真正补齐工作台身份证、缺失入口文件和入口规则。

怎么使用

A 测用户不需要懂 Core，先跑命令，看体检单即可。

检查当前目录

starwork doctor

检查指定目录

starwork doctor --target ~/Desktop/my-workspace

给 Agent 或脚本读取

starwork doctor --target ~/Desktop/my-workspace --json

显示所有通过项

starwork doctor --target ~/Desktop/my-workspace --verbose

当前不做什么

这些边界很重要，避免用户误以为 doctor 会替他动文件。

不自动修复

当前没有 --fix。Doctor 只检查和建议，不会静默创建、移动、删除或合并文件。

不判断内容质量

它不会评价文章、课程、业务材料好不好，只看结构和边界是否符合 StarWork。

不读取远程主库

它只检查本地工作区，不去检查云端权限、消息平台、GitHub 状态或主库实时内容。
