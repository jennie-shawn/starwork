StarWork / Product Shape / Business Model

让 Agent 真正进入长期工作，而不是只进入一次对话。

StarWork 的产品形态不是“又一个 Agent 壳”，也不是“文件夹模板”。它是一套跨 Agent 的 AI 工作系统：用 Core 建可信协议，用 CLI 降低启动门槛，用 Packs 交付场景价值，用 Course 完成教育、转化和高客单服务。

产品判断 不追底层 Runtime，借力 Codex、Claude Code、Cursor、Trae 等 Agent。
价值重心 把用户的长期工作变成 Agent 可读、可接、可审计的工作区系统。
商业抓手 Packs 交付场景解决方案，Course 承接教学、陪跑和组织级转化。

用户买的不是工具，是连续性

真实痛点不在“有没有 AI”，而在每次换 Agent、换会话、换项目时，用户都要重新解释背景、边界、产物位置和当前进度。

一句话商业洞察

StarWork 不是靠“更聪明的模型”收费，而是靠“让聪明模型能进入真实工作流”收费。

  • 降解释成本每次 Agent 进入项目，都先读同一套工作区事实。
  • 降混乱成本过程材料和正式事实源分开，避免草稿污染产品。
  • 降迁移成本用户不被单一 Agent 绑定，底层工具更新时仍能接力。
  • 升场景价值Pack 把内容创作、项目管理、团队中枢等工作流产品化。

四层产品形态

Core、CLI、Packs、Course 不是四个平行产品，而是一个从可信协议到商业交付的递进结构。

工作区交付形态

StarWork 的可见产物是工作台，但工作台不是模板目录，而是由 Kit、Pack、Adapter 共同组装出的可运行系统。

商业飞轮

开源 Core 带来信任与传播；CLI 降低试用门槛；Packs 把具体人群的工作流变成可付费方案；Course 把方法论和落地服务放大。

收入阶梯

早期不需要复杂账号、授权和平台系统。先用清楚的价值阶梯验证：免费可信入口、低门槛工具、场景付费、陪跑服务。

Core

免费 / 开源 / 传播

  • 协议文档
  • 参考 Kit
  • 标准与生态入口
  • 建立可信度

CLI

免费或低价工具层

  • 初始化工作区
  • 健康检查
  • Agent 适配
  • Pack 安装

Packs

核心商品 / 场景方案

  • 内容创作者 Pack
  • 模板与规则
  • Demo 工作区
  • 行业扩展包

Course

高客单 / 陪跑 / 组织落地

  • 方法论课程
  • 工作台搭建陪跑
  • 团队内训
  • 定制 Pack 咨询

早期商业主线：先用开源 Core 与 CLI 证明“可信且能跑”，再用 Content Creator Pack 证明“能解决具体工作”，最后把案例和方法论交给 Course 承接转化。

v0.1 优先级

第一个 Pack：内容创作者

首个商业验证不选抽象项目管理，而选内容创作者，因为它有清晰输入、生产链路、发布反馈和商业化线索。

市场进入路径

StarWork 的早期路径应该从“个人创作者能跑通”开始，再扩大到多项目个人工作台、小团队和组织化场景。

v0.1 到 v1.0

路线判断：Core 与 CLI 已经进入可验证状态，当前焦点应转向首个 Pack 与 Demo，而不是继续扩张地基。

定位边界

清楚说“不做什么”，能保护 StarWork 不被旧 Runtime 心智、平台幻想和过早工程复杂度拖偏。

StarWork 要做

  • 协议定义跨 Agent 工作区的最小可信结构。
  • 工具用 CLI 生成、检查、适配和安装 Pack。
  • 场景用 Packs 把真实工作流变成可复用产品。
  • 教育用 Course 帮用户真正改造自己的工作方式。

StarWork 暂不做

  • Runtime不自研完整 Agent Runtime，不追底层执行器。
  • Gateway第一阶段不做消息平台网关。
  • 绑定不默认依赖单一 Agent、SDK 或海外服务。
  • 平台v0.1 不提前设计复杂账号、授权和商业系统。
