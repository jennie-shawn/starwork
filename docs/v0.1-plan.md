# StarWork v0.1 计划

## 工作线

StarWork v0.1 采用三条线并行推进。

### Core v0.1

目标：冻结跨 Agent 工作区的最小协议。

当前状态：已形成中文第一阅读入口 `product/core/core-v0.1-protocol.md`，并已落地 `baseline + profiles + capabilities + presets + kits` 的维护结构。

关键问题：

- Agent 每次开始工作前读什么？
- 用户身份、项目状态、决策、教训和当前工作分别放在哪里？
- 一个 matter 如何创建、推进和归档？
- 哪些文件默认只读？
- 哪些动作需要用户确认？
- 多个 Agent 如何共享同一工作区上下文？

Core 必须保持跨行业、跨场景、跨 Agent，不混入自媒体内容创作者 Pack 的专属逻辑。

### CLI v0.1

目标：提供最小可用的安装和适配工具。

当前状态：已开始命令级设计，先从 `starwork init` 的友好交互流程开始。

首版命令边界：

```bash
starwork init
starwork doctor
starwork adapt
starwork pack install content-creator
```

CLI 在安装或更新时必须保护用户已有内容。它可以生成或更新 Agent 适配文件，但不能覆盖项目事实源。

### 自媒体内容创作者 Pack v0.1

目标：提供首个面向自媒体内容创作者的完整场景工作流包。

Pack 应覆盖：

- 账号定位
- 选题池
- 素材库
- 爆款拆解
- 内容日历
- 单篇内容事项
- 草稿与脚本
- 发布记录
- 数据复盘
- 用户反馈
- 商业化线索
- 风格库 / 表达偏好
- 平台规则备忘

最小闭环：

```text
灵感输入 -> 选题整理 -> 内容大纲 -> 正文/脚本 -> 发布记录 -> 数据复盘 -> 下一轮选题
```

验收标准：

- 普通内容创作者能看懂工作区。
- Agent 知道素材、选题、草稿、发布记录和复盘分别放哪里。
- 用户不需要理解 Core 内部机制也能开始工作。
- 项目能产出一个完整 Demo 案例。

## 首轮事项

首轮应建立以下事项：

1. Core v0.1 协议草案：已形成第一版协议入口
2. CLI v0.1 命令边界：进行中，当前聚焦 `starwork init`
3. 自媒体内容创作者 Pack v0.1 MVP 范围
4. 旧 Runtime spike 复核

这些事项中的成熟草案再晋升到 `product/`。
