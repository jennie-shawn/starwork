# StarWork v0.1 计划

## 工作线

StarWork v0.1 采用三条线并行推进。

### Core v0.1

目标：冻结跨 Agent 工作区的最小协议。

当前状态：已封版。已形成中文第一阅读入口 `product/core/core-v0.1-protocol.md`，并已落地 `baseline + profiles + capabilities + presets + kits` 的维护结构。Kit 结构已收敛为五类，事实依据见 `product/core/kits/kit-structure-reference.md`。

关键问题：

- Agent 每次开始工作前读什么？
- 用户身份、项目状态、决策、教训和当前工作分别放在哪里？
- 一个 matter 如何创建、推进和归档？
- 哪些文件默认只读？
- 哪些动作需要用户确认？
- 多个 Agent 如何共享同一工作区上下文？

Core 必须保持跨行业、跨场景、跨 Agent，不混入内容创作者 Pack 的专属逻辑。

### CLI v0.1

目标：提供最小可用的安装和适配工具。

当前状态：CLI v0.1 最小闭环已完成。`starwork init`、`starwork spawn`、`starwork doctor`、`starwork adapt` 和 `starwork pack install` 第一版均已落地。

首版命令边界：

```bash
starwork init
starwork spawn
starwork doctor
starwork adapt
starwork pack install content-creator
```

CLI 在安装、生成或更新时必须保护用户已有内容。它可以生成或更新 Agent 适配文件，也可以从 Hub 生成新项目工作台，但不能覆盖项目事实源。

公开 A 测状态：

- GitHub 仓库为 `jennie-shawn/starwork`。
- npm 包为 `@jennie-shawn/starwork`。
- `starworkInit` 与 `starworkSpawn` 已可通过 `npx skills add jennie-shawn/starwork` 安装。
- 本地产品仓库已准备 `0.1.0-alpha.3`，npm `latest` 仍需发布确认后从 `0.1.0-alpha.2` 更新。

M2 完成后新增的定制生成方向：

- `starwork spawn --blueprint` 第一版已落地，用于让 CLI 按工作台定制单生成定制化卫星项目。
- `starworkSpawn` skill 已落地第一版，用于帮助 Agent 生成 `blueprint.json`、项目规则和 seed 规划。
- 当前支持新增目录、路径覆盖、规则注入、seed 文件、dry-run 和 doctor 基础检查；暂不支持 `renames`、`removals` 和自动安装 Pack。

`init` 当前采用：

```text
工作区类型 -> Kit
Pack 选择 -> Pack 语言配置
Kit 文件 + Pack 规则 / seed / templates -> 最终工作台
```

### 内容创作者 Pack v0.1

目标：提供首个面向内容创作者的完整场景工作流包。

Pack 应覆盖：

- 账号定位
- 选题池
- 素材库
- 爆款拆解
- 内容日历
- 单篇内容事项
- 草稿与脚本
- 发布记录
- 数据复盘
- 用户反馈
- 商业化线索
- 风格库 / 表达偏好
- 平台规则备忘

Pack 源包结构已升级为语言无关业务角色 + 多语言落地配置：

```text
pack.json
languages/zh.json
languages/en.json
rules/<language>/
templates/<language>/
seed/<language>/
```

最小闭环：

```text
灵感输入 -> 选题整理 -> 内容大纲 -> 正文/脚本 -> 发布记录 -> 数据复盘 -> 下一轮选题
```

验收标准：

- 普通内容创作者能看懂工作区。
- Agent 知道素材、选题、草稿、发布记录和复盘分别放哪里。
- 用户不需要理解 Core 内部机制也能开始工作。
- 项目能产出一个完整 Demo 案例。
