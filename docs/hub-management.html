Hub + Satellite

多项目中枢管理机制

这页给第一次听 StarWork 多项目机制的人看：中枢不是一个巨大的项目文件夹，而是共享规则、知识、身份、技能和项目登记的管理层；真正的项目执行留在各自的卫星项目里。

Hub 管共享统一维护身份、教训、知识、skills、项目注册和跨项目联络。
Satellite 管执行每个具体项目维护自己的状态、事项、过程材料和正式成果。
Registry 管发现中枢只登记项目在哪里、是否活跃，不吞掉项目进度正文。
联络单管通信项目之间的请求、同步、依赖和交接，用联络单留痕。

一句话讲清楚

给不了解 StarWork 的人，可以先用这个版本解释，不需要一上来讲 Core、Pack、Preset。

多项目中枢就是一个“项目控制塔”：它不替每个项目干活，而是统一维护跨项目可复用的身份、经验、知识、skills 和项目登记；每个具体项目仍然在自己的工作台里独立推进。

三个角色

理解这套机制，只要先分清 Hub、Satellite、Registry 三件事。

01 / Hub

多项目中枢

维护跨项目共享的东西：身份、教训、知识、skills、项目注册表、跨项目联络和回写审核。它是共享机制层，不是某个业务项目本身。

02 / Satellite

卫星项目

每个具体项目自己的工作台。项目状态、当前工作、事项、资料、过程材料、正式成果，都留在这里独立维护。

03 / Registry

项目注册表

Hub 里的项目清单。它记录项目 ID、名称、路径、状态、Core 版本和同步元数据，但不保存项目进度正文。

为什么要分开

如果所有项目都塞进一个大目录，短期看方便，长期会让事实源和责任边界混在一起。

不分开会怎样

  • 每个项目都会复制一份身份、教训和规则，越来越不一致。
  • 共享知识和项目草稿混在一起，不知道哪里才是事实源。
  • 中枢会变成巨大的进度数据库，项目越多越难读。
  • 一个项目的临时判断容易污染所有项目的公共规则。

分开以后

  • Hub 只维护可复用、可审计、跨项目共享的内容。
  • Satellite 只维护项目自己的执行事实和产出。
  • 项目发现靠 registry，项目进度仍然留在项目里。
  • 可复用经验先进入回写审核，确认后再进入 Hub。

实际怎么工作

用户感知上是“先建中枢，再从中枢创建项目”。CLI 内部负责检查、生成、登记和同步元数据。

Step 1

创建 Hub

用 starwork init --type hub 建立多项目中枢。这里放共享身份、教训、知识、skills 和项目注册表。

Step 2

生成项目

用 starwork spawn 从 Hub 创建 Satellite。CLI 会复制快照、挂载共享资源，并写入同步元数据。

Step 3

项目内推进

Agent 在 Satellite 内维护项目状态、事项、资料和成果。Hub 只知道项目在哪里、是否活跃、如何同步。

starwork init --type hub --target ~/my-hub --yes

starwork spawn \
  --hub ~/my-hub \
  --name "内容产品官网" \
  --target ~/projects/content-site \
  --mode matter \
  --yes

starwork doctor --target ~/projects/content-site

哪些内容放哪里

这张表是解释边界最有用的部分：Hub 不是所有内容的收纳箱，Satellite 也不是孤岛。

内容 放在 Hub 放在 Satellite 原因
长期身份、偏好、工作方式 identity/ 初始化快照或只读参考 这是跨项目稳定资源，应由中枢统一维护。
跨项目教训 lessons/ 本项目候选教训 项目里先沉淀，确认可复用后再回写到 Hub。
共享知识库 知识/ 或 knowledge/ 只读链接 避免多个项目各自修改同一份公共知识。
共享 skills skills/ 软链接或挂载入口 让多个项目使用同一套能力，而不是复制出很多分叉。
项目状态和当前工作 只登记元数据 _系统/上下文/当前项目.md、_系统/任务/当前工作.md 项目事实必须留在项目工作台内。
项目产物、资料、事项 不存正文 输出/、参考资料/、事项/ 这些是具体项目的执行内容，不属于中枢公共层。
可复用更新建议 .incoming/ 审核后合并 发起候选回写 先审核再进入 Hub，避免项目临时经验污染公共资源。
项目之间的请求、同步和交接 项目/联络/ 或中枢联络入口 _系统/跨项目/ 联络单负责“项目和项目怎么说话”，不替代项目状态，也不直接改变 Hub 正式知识。

联络单机制

联络单是多项目协作里的“可追踪消息”。它解决项目之间怎么提请求、交接、同步和回复的问题。

它是什么

项目之间的正式留言

当项目 A 需要项目 B 提供信息、同步进展、确认边界、交接材料，或者向 Hub 说明一个跨项目事项时，就写联络单。它应该有明确发送方、接收方、背景、请求、截止时间和期望回复。

它不是什么

不是项目进度，也不是知识回写

联络单不存项目完整进度，项目进度仍在项目自己的状态和事项里；它也不直接把内容合并进 Hub 的身份、教训或知识。可复用内容仍应走 .incoming/ 审核。

发送

项目 A 写联络单

在本项目 _系统/跨项目/ 留一份发送记录，说明要找谁、为什么找、希望对方做什么。

→
路由

Hub 留痕和分发

Hub 的联络入口记录这次跨项目通信，帮助未来查到“哪个项目找过哪个项目”。

→
回复

项目 B 处理并回复

项目 B 在自己的工作台处理请求，并通过联络单回复。需要沉淀为公共知识时，再另走回写审核。

可以把 registry 理解成“通讯录”，把联络单理解成“正式邮件”，把 .incoming 理解成“投稿审核箱”。三者都和中枢有关，但职责完全不同。

容易误解的点

这几个边界要讲清楚，别人就不容易把 Hub 理解成“更大的项目文件夹”。

Hub 不是项目进度数据库

Hub 的 项目/registry.json 只回答“有哪些项目、在哪里、什么状态”。具体进度、事项、判断和产物仍在各自 Satellite。

Satellite 不是 Hub 的子目录

Satellite 是独立工作台，可以在任意路径。它通过 .core-sync.json 记录来源 Hub、共享资源和同步关系。

共享资源不是随便改

知识和 skills 默认来自 Hub。项目内如果产生可复用内容，应先走候选回写和审核，而不是直接改 Hub 正式内容。

spawn 不是 init 的替代品

init 创建当前工作台，spawn 从已有 Hub 创建并登记一个新项目。它们处理的是不同层级的问题。

联络单不是 registry

registry 记录项目元数据，联络单记录项目之间的一次正式沟通。不要把跨项目请求塞进项目注册表。

联络单不是 .incoming

.incoming 用来审核候选公共内容；联络单用来请求、同步、交接和回复。一次联络可能产生回写候选，但它们不是同一件事。

一句话版本

如果你要在会议里快速解释，可以直接用下面这段。

StarWork 的多项目机制采用 Hub + Satellite：Hub 是中枢，负责共享身份、教训、知识、skills、项目注册和联络入口；Satellite 是具体项目，负责自己的状态、事项、资料和成果。Hub 通过 registry 找到项目，通过 spawn 创建项目，通过联络单协调项目之间的请求和交接，通过 .incoming 接收候选回写，但不会把项目进度正文吞进中枢。
