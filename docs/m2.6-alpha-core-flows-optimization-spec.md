# M2.6 A 测核心链路优化 SPEC

## 状态

- 版本：v0.1 draft
- 所属阶段：M2.6 既有功能优化与 A 测稳定化
- 所属模块：StarWork CLI / Skills / Kits / Docs
- 当前 npm latest：`@jennie-shawn/starwork@0.1.0-alpha.13`
- 目标：把 A 测用户最容易接触到的四条核心链路打磨稳定，降低用户和 Agent 的误解、误装、误写风险。

## 一句话定义

M2.6 不新增大的产品面，而是把已经有的 StarWork 能力变成四条稳定路径：

```text
安装 StarWork
  -> 创建新工作台
  -> 诊断 / 升级历史模板
  -> 用 Hub 管理多个项目
```

每条路径都要做到：用户知道怎么开始，Agent 知道该读什么，CLI 只做可验证执行，Skill 负责判断但不静默写入。

## 背景

StarWork 已经完成：

- `init`
- `doctor`
- `upgrade`
- `spawn`
- `audit`
- `repair`
- `adapt`
- `pack install`
- `multiagent`
- 系统级 Skills：`starworkInit`、`starworkDoctor`、`starworkMultiagent`
- Kit 自带 Skills：Hub Kit 的 `starworkSpawn`、`starworkAudit`，Project Kit 的 `neat-freak`

现在的问题不是“缺功能”，而是 A 测用户真实使用时可能在这些地方卡住：

- 安装 CLI 和 Skills 时不知道该用哪条命令。
- Agent 不知道应该读哪份安装指南。
- `init` 交互问了不该问的问题，或者没问关键问题。
- 历史模板升级时，Agent 生成的 `AGENTS.md` 质量不稳定。
- Hub、Project、Kit 自带 Skill、Hub 托管 Skill 的边界容易混。
- 文档里的版本、阶段和能力描述容易落后于 npm 包。

所以 M2.6 的核心任务是“把已发布能力变顺”，不是继续扩张新命令。

## 非目标

- 不启动 Content Creator Pack v0.1 的大开发。
- 不设计 `starwork update`。
- 不设计复杂 skill 市场或云端 skill registry。
- 不引入账号、云同步、消息平台 gateway。
- 不把 `doctor` 变成自动决策器；AI 判断仍由 Skill 完成。
- 不把 Skill 直接变成文件写入执行器；写入仍交给 CLI 或用户确认后的工具调用。

## 核心链路一：安装链路

### 目标

A 测用户或其 Agent 能在 5 分钟内完成：

```text
安装 CLI
安装系统级 Skills
验证版本
验证 Skills 可见
知道下一步该 init 还是 doctor
```

### 当前标准命令

安装 CLI：

```bash
npm install -g @jennie-shawn/starwork
starwork --version
starwork --help
```

不全局安装 CLI：

```bash
npx @jennie-shawn/starwork --version
npx @jennie-shawn/starwork --help
```

安装系统级 Skills：

```bash
npx skills add jennie-shawn/StarWork -g -a codex -y
```

系统级 Skills 只应包含：

```text
starworkInit
starworkDoctor
starworkMultiagent
```

不应全局安装：

```text
starworkSpawn
starworkAudit
neat-freak
```

### 需要优化

1. README、A 测指南、Agent 安装指南的安装命令必须一致。
2. Agent 安装指南必须明确要求 Agent 读取 raw 文档，不引用本地路径。
3. `npx skills add` 安装结果需要在文档里有清晰验收方式。
4. 安装后应该给用户两个下一步分支：
   - 新建工作台：进入 `starworkInit`
   - 检查旧目录：进入 `starworkDoctor`
5. 发布后必须同步：
   - `product/package.json`
   - `product/README.md`
   - `product/docs/alpha-test-guide.md`
   - `product/docs/agent-install-guide.md`
   - `product/docs/cli-skill-registry.html`
   - `product/docs/roadmap.md`
   - `product/docs/roadmap.html`

### 验收标准

- 外部用户只看公开 README，也能安装 CLI。
- 用户把 Agent 指向 `agent-install-guide.md` raw 页面后，Agent 不会创建临时工作区，除非先询问用户。
- `npx skills add jennie-shawn/StarWork -g -a codex -y` 不会安装 Kit 自带 Skills。
- 文档中的 npm latest 与实际发布版本一致。

## 核心链路二：新建工作台链路

### 目标

让用户通过友好的 `starworkInit` 采访和稳定的 `starwork init` 执行，创建符合预期的 Project 或 Hub 工作台。

### 标准流程

```text
用户说明想建立工作台
  -> starworkInit 判断是 Project 还是 Hub
  -> 询问语言 zh / en
  -> Project 默认 general Pack
  -> Hub 默认 hub-management Pack
  -> 如用户需要，再进入定制工作台采访
  -> CLI dry-run
  -> 用户确认
  -> CLI 执行
  -> doctor 检查
```

### 当前分支规则

Hub：

- 不询问 Pack。
- 不询问 matter。
- 不默认创建项目执行层目录。
- 自动使用 Hub Kit + `hub-management` Pack。
- Kit 自带 `starworkSpawn`、`starworkAudit`。

Project：

- 默认使用 `general` Pack。
- 必须询问语言。
- 默认不展示未定稿场景 Pack。
- 如用户提出定制，当前只支持围绕通用 Pack 的目录和规则定制。
- Kit 自带 `neat-freak`。

### 需要优化

1. `starworkInit` 继续收紧采访问题，不出现当前没有 Pack 支撑的场景选项。
2. `init` 交互中的命名继续统一为 Project / Hub，不再回到旧 matter 类型。
3. `init --dry-run` 输出要让用户看懂会创建哪些目录、会安装哪些 Kit 自带 Skill。
4. `init` 完成后提示下一步：
   - Project：运行 `starwork doctor`，必要时 `starwork adapt`
   - Hub：运行 `starwork doctor`，再用 `starworkSpawn` / `starwork spawn` 创建项目
5. 对已有非空目录的处理继续保持保守，不覆盖用户文件。

### A 测反馈专项 SPEC

单项目初始化反馈拆成 5 个可独立实现和验收的 SPEC：

- [m2.6-project-init-feedback/README.md](m2.6-project-init-feedback/README.md)：专项索引，便于把 5 个反馈项整体交给实现 Agent。
- [m2.6-project-init-feedback-01-identity-spec.md](m2.6-project-init-feedback/m2.6-project-init-feedback-01-identity-spec.md)：修正独立 Project 的 identity 模板口径，避免混入 Hub / Satellite 身份同步描述。
- [m2.6-project-init-feedback-02-hub-sync-boundary-spec.md](m2.6-project-init-feedback/m2.6-project-init-feedback-02-hub-sync-boundary-spec.md)：收紧独立 Project 与 Hub / Satellite 同步目录边界，避免生成用户无法理解的主库同步文件。
- [m2.6-project-init-feedback-03-current-project-purity-spec.md](m2.6-project-init-feedback/m2.6-project-init-feedback-03-current-project-purity-spec.md)：保证 current-project 只记录项目事实，不写入初始化执行记录、blueprint 解释或 Agent 推理过程。
- [m2.6-project-init-feedback-04-system-placeholder-templates-spec.md](m2.6-project-init-feedback/m2.6-project-init-feedback-04-system-placeholder-templates-spec.md)：补齐 system 层占位模板，让用户和 AI 知道每类文件该写什么。
- [m2.6-project-init-feedback-05-agents-guardrails-spec.md](m2.6-project-init-feedback/m2.6-project-init-feedback-05-agents-guardrails-spec.md)：重写 AGENTS 入口规则和 Skill 防污染约束，明确启动读取时机，阻止 Skill 把过程话术写进项目规则。

### 验收标准

- 新用户能在 Project / Hub 两种类型之间做出选择。
- 中文用户和英文用户都能生成对应语言的 Project 工作台。
- Hub 路径仍保持英文协议路径，但 README / AGENTS 文案按语言输出。
- Project Kit 和 Hub Kit 自带 Skill 分发正确。
- `init -> doctor` 对 Project 和 Hub 都通过。
- 以上 5 个专项 SPEC 的验收标准全部通过后，单项目初始化链路才视为完成。

## 核心链路三：历史模板诊断与升级链路

### 目标

让已经使用旧模板或非标准目录的用户，能够安全接入 StarWork，而不是被要求重建工作台。

### 标准流程

```text
starwork doctor --json
  -> 暴露目录 inventory、signals、legacy 候选和 workspace facts
  -> starworkDoctor 判断当前目录与 StarWork Core 工作逻辑的贴近程度
  -> starworkDoctor 生成升级建议和 upgrade blueprint
  -> 用户确认
  -> starwork upgrade --blueprint
  -> starwork doctor 复查
```

### 边界

`doctor`：

- 只探测和暴露事实。
- 不输出 next step。
- 不判断用户该选哪个 Kit / Pack。
- 不自动生成升级计划。

`starworkDoctor`：

- 负责 AI 判断。
- 负责解释当前目录和 StarWork Core 逻辑的贴近程度。
- 负责提炼用户原有规则文档中的有效内容。
- 负责生成保守 upgrade blueprint。

`upgrade`：

- 只执行用户确认过的 blueprint。
- 保护已有文件。
- 不自行判断升级策略。

### 需要优化

1. `doctor --json` 继续增强目录语义信号，但不能加入建议型字段。
2. `starworkDoctor` 必须把用户原有规则文档提炼后融入升级方案。
3. 生成的 `AGENTS.md` 必须短、清晰、有边界，不出现内部占位符。
4. 详细规则进入 `.starwork/rules/`，`AGENTS.md` 只作为入口。
5. Hub-like 旧主库继续支持 preserve-names，不强制改成新版标准路径。
6. 旧 `_系统/跨项目/` 只作为 legacy local handoff 信号，不作为新标准建议。

### 验收标准

- 对旧模板运行 `doctor --json`，输出足够事实供 Agent 判断。
- `starworkDoctor` 输出的人话结论能让用户理解“现在目录像什么、差什么、怎么升级”。
- 升级后的 `AGENTS.md` 不乱、不长、不泄露内部占位符。
- 用户原有规则里的有效内容不会丢。
- `upgrade --dry-run` 能清楚展示将写入什么。
- `upgrade --yes` 后再跑 `doctor` 能通过或只剩可解释 warning。

## 核心链路四：Hub 多项目管理链路

### 目标

让 Hub 真正成为“多项目中枢”，而不是另一个项目工作台。

### 标准流程

```text
starwork init --type hub
  -> doctor 检查 Hub
  -> starworkSpawn 生成 spawn blueprint 或用户直接 starwork spawn
  -> spawn 创建 Project Satellite
  -> Hub projects/registry.json 登记项目
  -> starwork audit 巡检已登记项目
  -> starworkAudit 解释问题并生成 repair blueprint
  -> starwork repair 执行保守修复
```

### Hub 边界

Hub 维护：

- `projects/registry.json`
- `projects/coordination/`
- `.starwork/handoff/`
- `.incoming/`
- `identity/`
- `lessons/`
- `knowledge/`
- `skills/registry.json`
- `workspace/`

Hub 不维护：

- 单个项目的当前工作正文。
- 单个项目的项目状态正文。
- 卫星项目的业务成果。
- `_系统/任务/当前工作.md` 这类 Project 执行入口。

### Skill 分发边界

系统级 Skill：

```text
product/skills/
```

Kit 自带 Skill：

```text
product/kit-skills/
```

Hub 托管 Skill：

```text
<hub>/skills/
```

Project 本地 Skill：

```text
<project>/.agents/skills/
<project>/.claude/skills/
```

### 需要优化

1. `spawn` 不硬编码旧路径，继续从 Hub state 和语言 profile 读取。
2. `audit` 输出要区分 Hub 自身问题、registry 问题、Satellite 问题。
3. `repair` 必须保持 blueprint 执行器定位，不自动判断修复策略。
4. `starworkAudit` 是 Hub Kit 自带 Skill，不是系统级 Skill。
5. Hub 创建的 Satellite 要继承语言、知识链接和可选 Skill 分发策略。
6. Hub 文档需要持续解释：Hub 是中枢，不是大号项目文件夹。

### 验收标准

- `init --type hub -> doctor` 通过。
- `spawn -> doctor` 生成的 Project 通过。
- Hub `projects/registry.json` 正确记录项目 ID、路径和状态。
- `audit --hub <hub> --json` 能暴露所有项目健康事实。
- `repair --blueprint` 只执行明确修复，不越权改业务文档。
- `npx skills add` 不会安装 `starworkSpawn` 或 `starworkAudit`。

## 横向规则：文档同步

任何核心链路发生变化时，必须同步这些文档：

- `product/README.md`
- `product/docs/agent-install-guide.md`
- `product/docs/alpha-test-guide.md`
- `product/docs/cli-skill-registry.html`
- `product/docs/roadmap.md`
- `product/docs/roadmap.html`
- `_系统/上下文/current-projects.md`
- `_系统/任务/current-work.md`

同步原则：

- 版本号使用绝对版本，例如 `0.1.0-alpha.13`。
- 阶段使用明确名称，例如 `M2.6 既有功能优化与 A 测稳定化`。
- 不写“最近”“刚刚”“下一步可能”这类相对表达。
- 不在公开文档里写本地路径作为用户安装依据。

## 横向规则：测试与验证

每次改动至少考虑：

```bash
npm test
npm pack --dry-run
starwork --version
starwork --help
```

按链路补充手动验收：

- 安装链路：全局安装、npx 运行、系统 Skills 安装。
- 新建链路：Project / Hub 分别 `init -> doctor`。
- 升级链路：旧模板样本 `doctor --json -> starworkDoctor -> upgrade --dry-run`。
- Hub 链路：`hub init -> spawn -> audit -> repair dry-run`。

## 实施优先级

### P0：文档和版本一致性

状态：已完成第一轮落地。

- 校准当前版本、当前阶段和下一步。
- 确保 README、A 测指南、Agent 安装指南和注册表不互相矛盾。

### P1：历史模板升级质量

状态：已完成第一轮落地。

原因：这是现有用户平滑迁移的关键。

- 继续优化 `starworkDoctor` 的规则提炼。
- 用真实旧模板样本检查升级后的 `AGENTS.md`。
- 确保 `.starwork/rules/` 与 `AGENTS.md` 分工清楚。

### P2：安装和 init 体验

状态：已完成第一轮落地。

原因：这是新用户第一触点。

- 优化安装后的验证提示。
- 优化 `starworkInit` 的分支采访。
- 避免出现未定稿 Pack 选项。

### P3：Hub 管理链路

状态：已完成第一轮落地。

原因：这是高级用户和多项目用户的核心价值。

- 继续打磨 `spawn`、`audit`、`repair` 的信息结构和边界。
- 明确 Hub Kit 自带 Skill 与 Hub 托管 Skill 的差异。

### P4：multiagent 体验

状态：已完成第一轮落地。

原因：这是协作增强能力，重要但不应压过升级和安装链路。

- 优化 `status` 的人类可读输出。
- `release` 前提醒更新 worklog。
- `share` 后提示其他 lane 应读哪里。

## 开放问题

1. 是否需要给 A 测用户提供一个“一键自检脚本”，自动跑版本、help、系统 Skill 可见性和最小 init dry-run。
2. 是否需要在 `starworkDoctor` 中加入固定的 `AGENTS.md` 质量检查清单。
3. Hub `audit` 的人类可读输出是否需要独立做一版，而不是只依赖 JSON + Skill 解释。
4. `multiagent status` 是否需要默认输出摘要，`--json` 只给 Agent / 自动化使用。
5. A 测反馈是否要单独形成结构化 issue 模板或反馈表。
