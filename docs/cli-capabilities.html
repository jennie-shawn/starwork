StarWork CLI v0.1

CLI 能力说明

CLI 是 StarWork 的产品化入口。它负责创建工作台、检查结构、适配不同 AI 工具、组织多 AI 会话协作，并把场景能力安全安装进去；定制化生成由 spawn 定制方案和 starworkSpawn skill 共同完成第一版闭环。

init从空文件夹生成一个可工作的 StarWork 工作台。
spawn从多项目中枢生成并登记一个新项目工作台。
doctor检查工作台是否健康，找出缺失、错配和风险。
adapt为不同 AI 工具生成轻量入口，但不复制规则源。
multiagent按项目自定义职责位组织多个 AI 会话协作。
pack install在健康工作台上补装场景能力。

核心命令

v0.1 不做大而全的 CLI，而是先完成一条可靠路径：能初始化、能生成中枢项目、能检查、能适配、能组织多 AI 协作、能补装场景能力。

init

生成

把基础结构和场景能力组装成一个可直接使用的 StarWork 工作台。普通用户不需要理解内部配置。

  • 支持普通项目工作台和多项目管理中枢。
  • 交互默认推荐普通项目；单项目默认使用通用工作能力。
  • 写入前预览，不静默覆盖已有内容。
  • 生成 `.starwork/workspace.json`，供后续命令读取。
node cli/bin/starwork.js init --type project --pack general --target ./workspace --yes

spawn

派生

从健康的多项目中枢生成一个被管理的新项目工作台，并把它登记回 Hub 的项目注册表。

  • 当前保留项目工作台模式，旧的事项型结构已封存。
  • 复制 Hub 的身份和教训快照。
  • 软链接 Hub 的知识和 skills 入口。
  • 写入 .core-sync.json 和 Hub projects/registry.json。
node cli/bin/starwork.js spawn --hub ./hub --name "新项目" --target ./project --mode project --language zh --yes

doctor

检查

检查工作台是不是完整。它只报告问题，不默认修复文件，是后续适配和场景能力安装的守门员。

  • 检查工作台身份证、入口文件和基础结构文件。
  • 检查正式成果区、日常工作区、场景初始文件和模板。
  • 支持用户可读输出和 --json。
  • 健康返回退出码 0，有阻塞问题返回 1。
node cli/bin/starwork.js doctor --target ./workspace --json

adapt

适配

为具体 AI 工具生成轻量适配入口。它不复制一整套规则，只告诉 AI 去读 StarWork 的统一规则源。

  • 支持 codex、claude、cursor、trae、all。
  • Claude 生成 CLAUDE.md。
  • Cursor 生成 .cursor/rules/starwork.mdc。
  • 适配结果记录到工作台身份证。
node cli/bin/starwork.js adapt claude --target ./workspace --yes

multiagent

协作

为同一项目建立自定义 AI 职责位，绑定具体会话，并登记跨 lane 可读输出。

  • 不内置固定职责，lane 由项目场景定义。
  • 使用 _系统/协作/agent-lanes.md 作为分工索引。
  • 使用 _系统/协作/shared.md 登记跨 lane 共享输出。
  • 不写入任务系统或锁机制。
node cli/bin/starwork.js multiagent init --lanes research,writing,review --target ./workspace --yes

pack install

扩展

在已有健康工作台上补装场景能力。它让用户可以先从通用工作台开始，再逐步加入具体场景能力。

  • 安装前必须通过阻塞级 doctor 检查。
  • 创建场景目录、初始文件和模板。
  • 把场景规则写入 .starwork/rules/。
  • 更新工作台身份证，已安装场景能力不重复安装。
node cli/bin/starwork.js pack install content-creator --target ./workspace --yes

详细能力与执行步骤

这一段按命令拆开看：每个 CLI 到底能做什么、执行时经过哪些步骤、适合什么时候用。

01 / 生成工作台

starwork init

把基础结构和场景能力合成一个最终工作台。它是用户第一次接触 StarWork 的入口。

node cli/bin/starwork.js init \
  --type project \
  --pack general \
  --target ./workspace \
  --yes

具备的能力

  • 识别目标目录是否已经是 StarWork 工作台。
  • 根据 --type 选择普通项目或多项目中枢。
  • 根据 --pack 安装通用、内容创作者或中枢管理场景能力。
  • 生成 AGENTS.md、系统状态文件、工作目录、场景初始文件和模板。
  • 写入 .starwork/workspace.json，作为后续命令读取的工作台身份证。
  • 遇到已有用户文件时生成旁路文件，不静默覆盖。

执行步骤

  1. 解析参数和目标目录。
  2. 检查是否已经位于 StarWork 工作台内。
  3. 确定工作区类型、语言和场景能力。
  4. 加载基础结构文件和场景语言配置。
  5. 渲染场景规则、初始文件和模板。
  6. 生成写入计划并展示预览。
  7. 确认后写入文件和工作台身份证。
  8. 提示下一步运行 doctor 和 adapt。
02 / 生成中枢项目

starwork spawn

从 Hub 生成一个新项目工作台。用户面对的是“生成一个新项目”，内部使用项目工作台基础结构。

node cli/bin/starwork.js spawn \
  --hub ./hub \
  --name "内容产品官网" \
  --id content-site \
  --target ./content-site \
  --mode project \
  --language zh \
  --yes

具备的能力

  • 检查 Hub 是否健康，且必须是 hub 工作区。
  • 只允许写入不存在或空目录。
  • 按 --mode 选择项目工作台生成方式。
  • 写入项目自己的工作台身份证和同步说明。
  • 回写 Hub 项目注册表，拒绝重复 ID 或重复路径。
  • --blueprint 支持按工作台定制单创建目录、写入路径、注入规则和复制初始文件。

执行步骤

  1. 定位并检查 Hub 工作台。
  2. 检查目标目录是否可写。
  3. 复制项目基础结构。
  4. 同步身份、教训、知识和 skills 入口。
  5. 写入 .starwork/workspace.json 与 .core-sync.json。
  6. 更新 Hub projects/registry.json。
  7. 提示运行 doctor。
03 / 检查健康

starwork doctor

检查当前工作台是否完整。它是守门员，不修复，只报告。

node cli/bin/starwork.js doctor \
  --target ./workspace

node cli/bin/starwork.js doctor \
  --target ./workspace \
  --json

具备的能力

  • 定位 .starwork/workspace.json。
  • 检查工作台身份证格式、版本、工作区类型、语言和场景能力。
  • 检查必要入口：入口规则、项目状态、当前工作。
  • 检查正式事实源和业务工作区。
  • 检查基础结构源文件是否完整落地。
  • 检查场景路径、初始文件和模板是否存在。
  • 支持用户可读输出和 JSON 输出。

执行步骤

  1. 从目标目录向上查找 StarWork 工作台。
  2. 读取并解析工作台身份证。
  3. 按基础结构推导关键入口路径。
  4. 检查基础结构文件完整性。
  5. 读取已安装场景能力的语言配置。
  6. 检查场景目录、初始文件和模板。
  7. 汇总 pass、info、warn、fail。
  8. 没有阻塞问题返回退出码 0，有阻塞问题返回 1。
04 / 适配 AI 工具

starwork adapt

给不同 AI 工具生成轻量入口。它不另起一套规则，而是把 AI 引回 StarWork 统一规则源。

node cli/bin/starwork.js adapt claude \
  --target ./workspace \
  --yes

node cli/bin/starwork.js adapt --agent cursor \
  --target ./workspace \
  --yes

具备的能力

  • 支持 codex、claude、cursor、trae、all。
  • Codex 复用 AGENTS.md，只登记适配状态。
  • Claude Code 生成 CLAUDE.md。
  • Cursor 生成 .cursor/rules/starwork.mdc。
  • Trae 生成 .trae/rules/starwork.md。
  • 适配前检查工作台是否健康。
  • 适配结果写入工作台身份证的 adapters 字段。

执行步骤

  1. 定位 StarWork 工作台。
  2. 读取工作台身份证。
  3. 运行阻塞级 doctor 检查。
  4. 确定目标 AI 工具。
  5. 生成对应适配入口内容。
  6. 生成写入计划。
  7. 确认后写入适配文件。
  8. 更新工作台身份证。
05 / 多 AI 协作

starwork multiagent

为同一项目建立多个 AI 会话的职责位协作层。它管理 lane、会话绑定、工作记录和共享索引，不管理事项生命周期。

node cli/bin/starwork.js multiagent init \
  --lanes research,writing,review \
  --target ./workspace \
  --yes

node cli/bin/starwork.js multiagent bind research \
  --session codex:manual-research-1 \
  --target ./workspace \
  --yes

具备的能力

  • 初始化 _系统/协作/agent-lanes.md 和 shared.md。
  • 按项目实际职责新增 lane，不预设后端、前端或测试。
  • 将当前或手动指定的 session 绑定到 lane。
  • 释放 lane 绑定，保留历史工作记录。
  • 用 status --json 输出可被 AI 读取的分工状态。
  • 用 share 登记其他 lane 需要读取的输出。

执行步骤

  1. 定位 StarWork 工作台。
  2. 读取或创建多 AI 协作 Markdown 文件。
  3. 校验 lane ID 和安全相对路径。
  4. 按命令更新 lane 表或 shared outputs 表。
  5. 确认后写回 Markdown 事实源。
  6. 不创建任务看板。
06 / 安装场景能力

starwork pack install

在已有健康工作台上补装场景能力。适合先建通用工作台，再加入内容创作者等场景能力。

node cli/bin/starwork.js pack install content-creator \
  --target ./workspace \
  --yes

具备的能力

  • 安装前运行阻塞级 doctor 检查。
  • 读取场景能力的 pack.json 和 languages/<language>.json。
  • 校验场景能力是否支持当前工作区类型。
  • 创建场景目录。
  • 写入场景初始文件和模板。
  • 把场景规则写入 .starwork/rules/。
  • 更新工作台身份证中的 packs 和路径配置。
  • 已安装场景能力不重复安装。

执行步骤

  1. 定位 StarWork 工作台。
  2. 读取工作台身份证。
  3. 确认场景能力尚未安装。
  4. 运行阻塞级 doctor 检查。
  5. 加载场景能力与语言配置。
  6. 生成目录、seed、templates、规则追加计划。
  7. 确认后安全写入。
  8. 更新工作台身份证，并建议重新运行 doctor。

用户流程

这套 CLI 的核心价值，是把协议和模板变成一条用户能走完的路。

第一次使用

  • 用户运行 init，选择工作区类型和场景能力。
  • CLI 生成基础结构 + 场景能力的最终工作台。
  • 用户运行 doctor，确认结构健康。
  • 用户运行 adapt，为当前 AI 工具生成入口。
  • 用户开始真实工作。

中枢创建项目

  • 用户先用 init --type hub 创建多项目中枢。
  • 用户运行 spawn，从中枢生成一个新项目工作台。
  • CLI 写入项目自己的工作台身份证和同步说明。
  • CLI 回写 Hub 的项目注册表。
  • 用户进入新项目运行 doctor。

典型脚本

node cli/bin/starwork.js init \
  --type project \
  --pack general \
  --target ./creator-workspace \
  --yes

node cli/bin/starwork.js doctor --target ./creator-workspace

node cli/bin/starwork.js adapt claude --target ./creator-workspace --yes

node cli/bin/starwork.js pack install content-creator \
  --target ./creator-workspace \
  --yes

现在怎么测试

建议先用临时目录完整跑一遍，不要在真实项目里第一次试。下面的命令都在 StarWork 仓库根目录下执行。

方式一：跑自动测试

这是最快确认 CLI 没坏的方法。当前测试覆盖 init、spawn、doctor、adapt、pack install 的核心路径。

cd /path/to/starwork
npm test

预期结果：50 个测试全部通过。

方式二：手动跑完整链路

这更接近真实用户体验，会在临时目录生成一个工作台，然后检查、适配、补装场景能力。

cd /path/to/starwork

TEST_DIR=/tmp/starwork-cli-manual-test
rm -rf "$TEST_DIR"

node cli/bin/starwork.js init \
  --type project \
  --pack general \
  --target "$TEST_DIR" \
  --yes

node cli/bin/starwork.js doctor --target "$TEST_DIR"

node cli/bin/starwork.js adapt claude --target "$TEST_DIR" --yes

node cli/bin/starwork.js pack install content-creator \
  --target "$TEST_DIR" \
  --yes

node cli/bin/starwork.js doctor --target "$TEST_DIR"

方式三：手动测试 spawn

这会先生成一个 Hub，再从 Hub 生成一个项目项目工作台，并检查两边是否健康。

cd /path/to/starwork

HUB_DIR=/tmp/starwork-hub-test
PROJECT_DIR=/tmp/starwork-spawn-test
rm -rf "$HUB_DIR" "$PROJECT_DIR"

node cli/bin/starwork.js init \
  --type hub \
  --target "$HUB_DIR" \
  --yes

node cli/bin/starwork.js spawn \
  --hub "$HUB_DIR" \
  --name "内容产品官网" \
  --id content-site \
  --target "$PROJECT_DIR" \
  --mode project \
  --language zh \
  --yes

node cli/bin/starwork.js doctor --target "$PROJECT_DIR"

方式四：手动测试 blueprint

这会用工作台定制单生成一个带定制目录和规则的卫星项目。

cd /path/to/starwork

BLUEPRINT_DIR=/tmp/starwork-blueprint-test
HUB_DIR=/tmp/starwork-blueprint-hub
PROJECT_DIR=/tmp/starwork-blueprint-project
rm -rf "$BLUEPRINT_DIR" "$HUB_DIR" "$PROJECT_DIR"
mkdir -p "$BLUEPRINT_DIR/rules"

cat > "$BLUEPRINT_DIR/rules/file-boundaries.md" <<'EOF'
正式成果放在 {{paths.formal_source}}。
EOF

cat > "$BLUEPRINT_DIR/blueprint.json" <<'EOF'
{
  "schema": "starwork.spawn_blueprint.v0.1",
  "name": "定制项目",
  "project_id": "custom-project",
  "base": { "mode": "project", "kit": "project", "language": "zh" },
  "paths": {
    "formal_source": "交付物/确认版本/",
    "business_work_area": "事项/"
  },
  "folders": ["资料库/", "交付物/确认版本/"],
  "agent_rules": [
    { "slot": "project.file_boundaries", "from": "rules/file-boundaries.md" }
  ]
}
EOF

node cli/bin/starwork.js init \
  --type hub \
  --target "$HUB_DIR" \
  --yes

node cli/bin/starwork.js spawn \
  --hub "$HUB_DIR" \
  --target "$PROJECT_DIR" \
  --blueprint "$BLUEPRINT_DIR/blueprint.json" \
  --yes

node cli/bin/starwork.js doctor --target "$PROJECT_DIR"

应该看到什么

  • doctor 最后输出 这个工作台结构完整，可以继续使用。
  • 目录里出现 AGENTS.md、CLAUDE.md、.starwork/workspace.json。
  • 补装内容创作者场景能力后出现 选题池/、素材库/、草稿与脚本/、发布记录/ 等目录。
  • .starwork/packs/content-creator/templates/ 下有内容模板。

故障测试

可以故意删掉关键文件，看 doctor 是否能发现问题。

rm "$TEST_DIR/AGENTS.md"
node cli/bin/starwork.js doctor --target "$TEST_DIR"

预期结果：退出码为 1，并提示 core.entry_rules.exists 失败。

边界

v0.1 的 CLI 是最小闭环，不是平台，不是账号系统，也不是 AI Runtime。

会做 不做
生成工作台结构、写入工作台身份证。 不处理账号、登录、授权、云端同步。
检查基础结构和场景能力是否一致。 不默认自动修复、不静默覆盖用户内容。
生成 AI 工具适配入口。 不为不同 AI 工具维护多套规则事实源。
补装场景能力并更新规则、模板和状态。 不做 Pack 市场、卸载、升级和复杂冲突迁移。
