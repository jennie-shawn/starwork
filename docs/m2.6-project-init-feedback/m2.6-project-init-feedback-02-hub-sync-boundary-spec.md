# M2.6 Project Init 反馈 02：Hub Sync 可见边界 SPEC

## 状态

- 版本：v0.1 implemented
- 所属阶段：M2.6 既有功能优化与 A 测稳定化
- 所属模块：StarWork Core / Project Kit / CLI init / CLI spawn
- 反馈来源：单项目初始化 A 测
- 问题类型：Core 架构和目录边界错误

## 用户反馈

单项目 `_system/` 下出现了 `main-repo-sync/`。用户不知道 Main Repo Sync 是什么，也没有建立 Hub，导致单项目工作台第一屏出现多项目架构概念。

## 根因

Project Kit 被设计为独立 Project 和 Satellite Project 共用，但模板把 Hub/Satellite 同步说明作为默认可见目录放进了所有 Project。

这会造成三个问题：

- 单项目用户被迫理解 Hub / main repo / sync。
- Agent 可能误以为当前项目已经接入 Hub。
- `_system/` 从“项目工作入口”变成“机制说明堆放处”。

## 目标

独立 Project 初始化结果不能默认出现 Hub 同步目录。

Hub 同步说明只在这些情况出现：

- `starwork spawn` 从 Hub 生成 Satellite Project。
- `starwork upgrade` 明确将旧主库或旧多项目工作区接入 Hub-like 模式。
- 用户显式要求把 Project 接入 Hub。

## 非目标

- 不取消 Hub / Satellite 能力。
- 不删除 `.starwork/sync.json` 作为机制运行层的可能性。
- 不在本 SPEC 里设计完整 `starwork attach-hub` 命令。
- 不让独立 Project 默认理解或维护 Hub registry。

## 目录分层决策

### 独立 Project 默认结构

独立 Project 可以有：

```text
_system/context/current-project.md
_system/tasks/current-work.md
_system/identity/README.md
_system/lessons/README.md
knowledge/README.md
.starwork/workspace.json
.starwork/skills.json
.starwork/handoff/state.json
```

独立 Project 默认不应有：

```text
_system/main-repo-sync/
_系统/主库同步/
.core-sync.json
.internal/
```

说明：

- `.starwork/handoff/` 可作为 StarWork 机制层保留，但 AGENTS 不应要求用户理解跨项目联络，除非已连接 Hub。
- `.core-sync.json` 是 legacy mirror，只应出现在 Satellite 或升级兼容场景。
- `.internal/` 是 Hub/主库协议快照，不应作为独立 Project 默认可见结构。

### Satellite Project 结构

从 Hub 生成的 Satellite 可以有：

```text
_system/main-repo-sync/README.md
.starwork/sync.json
.core-sync.json
.starwork/internal/
knowledge/ -> Hub knowledge
```

AGENTS 需要说明：只有涉及 Hub 资源、共享 skills 或跨项目交接时才读 main-repo-sync。

## 举一反三排查

实现时必须排查所有“独立 Project 默认出现 Hub 机制”的路径：

- `product/core/kits/project/_系统/主库同步/`
- `product/core/kits/project/.internal/`
- `product/core/kits/project/.starwork/handoff/`
- `product/core/kits/project/知识/README.md`
- `product/core/kits/project/README.md`
- `product/core/kits/project/AGENTS.md`
- `product/cli/src/cli.js` 的 `mapKitRelativePathForLanguage()`
- `product/cli/src/cli.js` 的 `renderEnglishMainRepoSyncReadme()`
- `product/cli/src/cli.js` 的 `buildInitPlan()` 是否复制所有 Kit 文件
- `product/cli/src/cli.js` 的 `buildSpawnPlan()` 是否单独补 Satellite 同步文件
- `product/core/kits/two-kit-architecture-spec.md` 中 Project Kit 目标结构是否仍把 Hub 同步目录作为默认项

排查关键词：

```text
main-repo-sync
主库同步
.core-sync.json
.internal
Hub sync
Satellite
main repository
跨项目联络
```

## 优化方案

### 方案 A：模板拆分

将 Project Kit 中的 Hub 同步相关文件移出默认 Kit，改为 Satellite overlay。

建议结构：

```text
core/kits/project/
core/kits/project-satellite-overlay/
```

`starwork init --type project` 只复制 `project/`。

`starwork spawn` 复制 `project/` 后，再叠加 `project-satellite-overlay/`。

### 方案 B：CLI 过滤

短期可在 `buildInitPlan()` 中过滤 Hub 同步文件：

```text
_系统/主库同步/**
_system/main-repo-sync/**
.core-sync.json
.internal/**
```

同时保留 `buildSpawnPlan()` 对 Satellite 的生成。

方案 A 更干净，方案 B 可作为 alpha 热修。

## 验收标准

### 独立 Project 验收

执行：

```bash
starwork init --type project --language en --pack general --target <tmp> --yes
starwork doctor --target <tmp>
```

必须满足：

- 不存在 `_system/main-repo-sync/`。
- 不存在 `.core-sync.json`。
- 不存在 `.internal/`。
- `AGENTS.md` 不出现 main repo / Hub sync 作为默认读取项。
- `doctor` 通过。

中文同理：

- 不存在 `_系统/主库同步/`。
- 不存在 `.core-sync.json`。
- 不存在 `.internal/`。

### Satellite 验收

执行：

```bash
starwork init --type hub --language en --target <hub> --yes
starwork spawn --hub <hub> --name Demo --target <project> --language en --yes
starwork doctor --target <project>
```

必须满足：

- 存在 `_system/main-repo-sync/README.md` 或等价 Satellite 同步说明。
- 存在 `.starwork/sync.json`。
- `.core-sync.json` 仅作为 legacy mirror 出现。
- `AGENTS.md` 说明 main-repo-sync 只在 Hub 资源、共享 skills 或跨项目交接时读取。
- `doctor` 通过。

### 防复发验收

- 测试必须同时覆盖 `init project` 和 `spawn satellite`。
- 测试必须断言独立 Project 不含 Hub 同步目录。
- 测试必须断言 Satellite 含 Hub 同步说明。
- `doctor` 不得要求独立 Project 具备 Hub 同步文件。
- 文档和 CLI help 不得把单项目称为 main repo 或 satellite。
